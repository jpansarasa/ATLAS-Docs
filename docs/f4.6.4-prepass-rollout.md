# F4.6.4 Entity-Resolution Pre-pass — Production Rollout Runbook

Audience: operator running the kill-switch flip after Phase 1 (F4.6.4.1 – F4.6.4.5) has merged.

Scope: enabling the SentinelCollector qualitative-extraction pre-pass that grounds prompts in OpenFIGI/EDGAR/NAICS-resolved entities. The pre-pass calls the spaCy NER sidecar for ORG candidates, hands them to SecMaster's `/api/resolve-entities` composer, and renders the result into the `## RESOLVED_ENTITIES` prompt block.

The implementation is deployed in production behind a kill-switch. The default state after Phase 1 merge is **flag-off**, identical observable behaviour to pre-F4.6.4 production. This document describes the operator-driven flip.

## Pre-flight (do NOT skip)

A1 — verify the kill-switch is off in the deployed config. The flip
mechanism is `sentinel_use_entity_resolution_prepass` in
`deployment/ansible/group_vars/all.yml`, which `compose.yaml.j2` renders
into the `Extraction__UseEntityResolutionPrepass` env var on the
`sentinel-collector` container:

```sh
sudo nerdctl exec sentinel-collector printenv \
  | grep -i UseEntityResolutionPrepass
# expected: Extraction__UseEntityResolutionPrepass=false
```

A2 — verify SecMaster `/api/resolve-entities` is healthy:

```sh
sudo nerdctl exec secmaster curl -sS \
  -X POST http://localhost:8080/api/resolve-entities \
  -H 'Content-Type: application/json' \
  -d '{"candidates":[{"surface":"Apple Inc.","kind":"CompanyName"}],
       "articleContext":"Apple Inc. reported earnings today."}' \
  | head -c 500
# expected: 200 OK with a ResolveEntitiesResponse JSON body
```

A3 — verify the spaCy NER sidecar responds (note: sidecar listens on `:3110`,
endpoint is `POST /ner`, wire shape matches `SpacyNerRequest`/`SpacyNerResponse`
in `SentinelCollector/src/Services/SpacyNerClient.cs`):

```sh
sudo nerdctl exec spacy-ner curl -sS \
  -X POST http://localhost:3110/ner \
  -H 'Content-Type: application/json' \
  -d '{"text":"Apple Inc. reported earnings today.","max_candidates":10,"min_confidence":0.8}' \
  | head -c 500
# expected: 200 OK with a candidates array, e.g.
# {"candidates":[{"surface":"Apple Inc.","kind":"CompanyName","score":0.85}]}
```

A4 — open the **ATLAS Entity Resolution (F4.6.4)** dashboard
(`/d/atlas-entity-resolution`). All panels should read **No data** —
that is the expected idle state with the flag off. If any panel reads
non-zero traffic, somebody already flipped the flag; do NOT proceed
until you understand why.

A5 — run the A/B audit harness from F4.6.4.4 to verify the treatment
arm beats control on the canary acceptance criteria. The harness lives
in `scripts/sentinel-quality-check/`:

```sh
# Step 1: produce a control + treatment pair of JSONLs.
# Output is a directory (control.jsonl + treatment.jsonl + index.json),
# NOT a single file; --limit/--seed mirror the plan's stratified 50-row sample.
python3 scripts/sentinel-quality-check/compare_base_vs_resolved.py \
  --limit 50 \
  --seed 42 \
  --output-dir /opt/ai-inference/training-data/f4.6.4-ab

# Step 2: grade the pair and emit the dated Markdown scorecard.
# The scorecard writes f4.6.4-ab-YYYY-MM-DD.md alongside an optional JSON sidecar.
python3 scripts/sentinel-quality-check/ab_scorecard.py \
  --input-dir /opt/ai-inference/training-data/f4.6.4-ab \
  --emit-json
```

Acceptance gates (from §A/B test design in the Phase 1 spec):

| Gate | Threshold | Action if missed |
|---|---|---|
| Subset A MAJOR rate (Treatment) ≤ Control − 3pp | minimum | flag stays off; root-cause |
| Subset A MAJOR rate (Treatment) ≤ 1% | stretch | proceed regardless if minimum met |
| Subset B MAJOR rate (Treatment) ≤ Control + 1pp | required | flag stays off; investigate regression |
| Overall MAJOR rate (Treatment) ≤ 4% | required | flag stays off |
| `extraction_confidence` mean within 0.05 of Control | required | investigate prompt drift |

**Do not flip the flag unless all "required" gates pass.** The minimum subset-A delta determines the rollout posture: meeting the stretch goal authorises the flip; meeting only the minimum means hold and re-evaluate (Phase 1 has no fractional ramp — flip is 0% → 100%).

## Flip (Day 0)

Phase 1 ships a single binary kill-switch — no hash-keyed canary-pct
mechanic exists in code. Rollout is a single-shot 100% flip gated on the
A/B scorecard above plus a small initial smoke window. (Granular % ramp
can be added in a follow-up story if operator experience demands it.)

Edit `deployment/ansible/group_vars/all.yml` and flip the var to `true`:

```yaml
# F4.6.4 prepass kill-switch — flip to true after A/B sign-off.
sentinel_use_entity_resolution_prepass: true
```

`compose.yaml.j2` renders this into the
`Extraction__UseEntityResolutionPrepass` env var on `sentinel-collector`.

Deploy:

```sh
ansible-playbook -i deployment/ansible/inventory/hosts \
  deployment/ansible/playbooks/deploy.yml --tags sentinel
```

Watch the **ATLAS Entity Resolution (F4.6.4)** dashboard for the first hour:

- **Prompt-grounding rate** should rise from 0% to a non-zero steady state
  (Phase 1 target: roughly 30-60% of qualitative articles surface at least
  one high-confidence entity). A flat 0% means the renderer wired in
  F4.6.4.3 isn't firing — roll back and investigate code, not config.
- **SecMaster-unavailable rate** should stay at 0%. Any nonzero value
  means the composer endpoint is unreachable from SentinelCollector;
  check container networking and SecMaster health.
- **Regex-fallback rate** should stay below 5%. A higher value points
  at spaCy sidecar instability — that is contingency, not steady state.
- **Pre-pass p95 latency** should sit well under 2s. Hard timeout is
  5s; a p95 trending toward it is the warning shot for the qualitative
  pipeline's overall SLO.
- **OpenFIGI cold-call rate** in the first hour will spike as the cache
  populates. After ~6h the cache hit ratio should settle ≥ 50%.

Rollback at any point: set
`sentinel_use_entity_resolution_prepass: false` and redeploy with
`--tags sentinel`. The next batch boundary picks up the new flag value;
no data migration is required.

## Sign-off (Day 3)

After 72 hours at 100% with the dashboard showing:

- SecMaster-unavailable rate sustained at 0%
- Regex-fallback rate sustained < 5%
- Pre-pass p95 < 2s
- OpenFIGI cache hit ratio ≥ 70%

…re-run `scripts/sentinel-quality-check/ab_scorecard.py` against the production
window to confirm the MAJOR-rate improvement persists in the wild.
File the scorecard under `docs/audits/` and close the F4.6.4 epic.

## Failure modes and response

| Symptom | Likely cause | First action |
|---|---|---|
| Prompt-grounding rate 0% with flag on | renderer wiring broken (F4.6.4.3) | roll back flag; bisect commits between F4.6.4.3 merge and current tip |
| Steady SecMaster-unavailable spike | composer endpoint 5xx or unreachable | check SecMaster `secmaster_entity_resolution_requests_total{outcome="error"}` rate; review SecMaster logs around the spike timestamp |
| Steady regex-fallback rate > 25% | spaCy sidecar unhealthy | check sidecar container status and `sentinel_spacy_ner_requests_total{outcome!="ok"}` breakdown on the dashboard |
| Pre-pass p99 climbing toward 5s | network or composer load | inspect `secmaster_entity_resolution_duration_ms_milliseconds_bucket` p99 — a gap between client-side and server-side latency points at the network/HTTP-pool seam |
| OpenFIGI cold-call rate climbing | cache TTL too short or NER candidate stream unbounded | inspect `secmaster_openfigi_cache_requests_total` breakdown; if `negative_hit` share climbs, NER is producing FPs — tighten candidate extraction before raising TTL |
| LLM extraction MAJOR rate regresses on subset B | grounding block confusing the model on no-entity content | roll back flag; the prompt-template change in F4.6.4.3 may need a tighter "(none)" fallback wording |

## What is **not** covered by this rollout

- Foreign-listed instruments: the EDGAR step short-circuits; entities
  resolve at `openfigi_no_us` and are dropped from the prompt by design.
  Phase 2 handles these via the existing `GeminiResolverClient`.
- Backfill of resolved-entity annotations onto the existing
  `extracted_observations` corpus: forward-only at MVP. Backfill is
  scoped to a separate ~0.5d story to run after this rollout completes
  and the A/B scorecard signs off.
- spaCy sidecar self-exposed metrics: the Phase 1 boundary metrics
  live on the C# side (`sentinel_spacy_ner_*`) which is sufficient for
  operator visibility. Adding `/metrics` on the Python sidecar is
  deferred to a follow-up alongside other sidecars (trafilatura,
  sandbox-kernel, sandbox-manager) to avoid double-counting the same
  call at internal and boundary layers.
