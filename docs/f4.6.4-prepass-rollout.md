# F4.6.4 Entity-Resolution Pre-pass — Operations Reference

Audience: operator inspecting / debugging the SentinelCollector qualitative-extraction pre-pass.

Scope: the prepass calls the spaCy NER sidecar for ORG candidates, hands them to SecMaster's `/api/resolve-entities` composer, and renders the result into the `## RESOLVED_ENTITIES` prompt block. (It originally also threaded the resolved-entity bag into the V2 semantic verifier's Stage 1.5; that verifier was removed as dead code in PR #647 — the prompt-block path below is what remains in production.)

The prepass runs **unconditionally** on every qualitative-dispatch row and every V2-production row. The original `Extraction__UseEntityResolutionPrepass` kill-switch was removed 2026-05-27 per the PoC "no feature flags" directive. Rollback is now ZFS snapshot + `git revert` + `ansible-playbook ... --tags sentinel-collector`.

## Health checks

H1 — SecMaster `/api/resolve-entities` is healthy:

```sh
sudo nerdctl exec secmaster curl -sS \
  -X POST http://localhost:8080/api/resolve-entities \
  -H 'Content-Type: application/json' \
  -d '{"candidates":[{"surface":"Apple Inc.","kind":"CompanyName"}],
       "articleContext":"Apple Inc. reported earnings today."}' \
  | head -c 500
# expected: 200 OK with a ResolveEntitiesResponse JSON body
```

H2 — spaCy NER sidecar responds (note: sidecar listens on `:3110`, endpoint is `POST /ner`, wire shape matches `SpacyNerRequest`/`SpacyNerResponse` in `SentinelCollector/src/Services/SpacyNerClient.cs`):

```sh
sudo nerdctl exec spacy-ner curl -sS \
  -X POST http://localhost:3110/ner \
  -H 'Content-Type: application/json' \
  -d '{"text":"Apple Inc. reported earnings today.","max_candidates":10,"min_confidence":0.8}' \
  | head -c 500
# expected: 200 OK with a candidates array, e.g.
# {"candidates":[{"surface":"Apple Inc.","kind":"CompanyName","score":0.85}]}
```

H3 — open the **ATLAS Entity Resolution (F4.6.4)** dashboard
(`/d/atlas-entity-resolution`). Steady-state expectations:

- **Prompt-grounding rate** non-zero (target: roughly 30-60% of qualitative
  articles surface at least one high-confidence entity). A flat 0% means
  the renderer wired in F4.6.4.3 isn't firing — investigate code, not
  config.
- **SecMaster-unavailable rate** at 0%. Any nonzero value means the
  composer endpoint is unreachable from SentinelCollector; check
  container networking and SecMaster health.
- **Regex-fallback rate** below 5%. A higher value points at spaCy
  sidecar instability — contingency, not steady state.
- **Pre-pass p95 latency** well under 2s. Hard timeout is 5s; a p95
  trending toward it is the warning shot for the qualitative pipeline's
  overall SLO.
- **OpenFIGI cold-call rate**: cache hit ratio should settle ≥ 50% after
  warm-up.

## Failure modes and response

| Symptom | Likely cause | First action |
|---|---|---|
| Prompt-grounding rate 0% | renderer wiring broken (F4.6.4.3) | ZFS snapshot + bisect commits between F4.6.4.3 merge and current tip |
| Steady SecMaster-unavailable spike | composer endpoint 5xx or unreachable | check SecMaster `secmaster_entity_resolution_requests_total{outcome="error"}` rate; review SecMaster logs around the spike timestamp |
| Steady regex-fallback rate > 25% | spaCy sidecar unhealthy | check sidecar container status and `sentinel_spacy_ner_requests_total{outcome!="ok"}` breakdown on the dashboard |
| Pre-pass p99 climbing toward 5s | network or composer load | inspect `secmaster_entity_resolution_duration_ms_milliseconds_bucket` p99 — a gap between client-side and server-side latency points at the network/HTTP-pool seam |
| OpenFIGI cold-call rate climbing | cache TTL too short or NER candidate stream unbounded | inspect `secmaster_openfigi_cache_requests_total` breakdown; if `negative_hit` share climbs, NER is producing FPs — tighten candidate extraction before raising TTL |
| LLM extraction MAJOR rate regresses on no-entity content | grounding block confusing the model | check the renderer's `(none)` fallback wording in `ResolvedEntitiesRenderer` |

## What is **not** covered by this component

- Foreign-listed instruments: the EDGAR step short-circuits; entities
  resolve at `openfigi_no_us` and are dropped from the prompt by design.
  Phase 2 handles these via the existing `GeminiResolverClient`.
- Backfill of resolved-entity annotations onto the existing
  `extracted_observations` corpus: forward-only at MVP.
- spaCy sidecar self-exposed metrics: the boundary metrics live on the
  C# side (`sentinel_spacy_ner_*`) which is sufficient for operator
  visibility. Adding `/metrics` on the Python sidecar is deferred
  alongside other sidecars (trafilatura, sandbox-kernel, sandbox-manager)
  to avoid double-counting the same call at internal and boundary layers.
